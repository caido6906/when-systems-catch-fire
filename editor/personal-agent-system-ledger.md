# 个人 Agent 写作系统成长账本

> 版本：2026-06-07 / 成长账本 v1.0
> 目的：无论点火框架理论是否最终成立，记录该项目已激活的个人生产系统能力

---

## 为什么需要这个账本

点火项目有两个产出：

1. **理论产出**：点火框架本身——能否成立还在验证中
2. **系统产出**：一套可运行的 Agent 辅助思想生产系统——已经成立

这个账本记录的是系统产出。它不应该被理论成败绑架。

## 一、Agent 实际承担的任务清单

### 资料整理与归档
- 从 Get笔记「我的写作风格」知识库批量导出 31 条笔记到 GitHub（`zy-writing-style-vault/corpus/raw/`）
- 处理 Unicode 文件名清洗、重复标题重命名、YAML frontmatter 标准化
- 从 Get笔记「好奇宝宝」知识库扫描 117 条笔记，识别并移除 51 条链接型笔记

### 概念收缩与边界建立
- 建立 16 个核心概念的四栏操作手册（定义/可观察证据/反证迹象/误判边界）
- 写出点火框架的边界声明：解释什么、不解释什么、边界判据
- 压缩为 7 条可检验命题，每条附证伪条件

### 反例生成与压力测试
- 建立三类可能推翻框架的反例及处理流程
- 对现有 72 条案例做萝卜坑风险审计（6 项风险发现）
- 建立冷酷反例账本（8 条条目，禁止辩护）

### 竞争理论对照
- 与 8 种相邻理论逐项对照（博弈论、创新扩散、制度理论、社会认同、Hirschman、组织文化、路径依赖、集体行动）
- 每项对照包含增量解释力论证和自我审视

### 编码工具建设
- 升级 8 格事件条件矩阵为可操作编码工具（含完整编码字段和规则表）
- 建立 12 个核心案例的统一模板

### 仓库维护
- 建立 `theory/` 理论目录（10 个文件）
- 建立 `theory/white-paper-0.1/` 白皮书框架（9 个文件，仅框架）
- 更新 `AGENT_ENTRY.md` 加入 Agent 工作宪章
- 创建 `CHANGELOG.md`
- 维护 `LEARNINGS.md`（经验教训库，8 条条目）
- 跨知识库链接笔记迁移（76 条笔记在 4 个知识库之间的精确操作）

### 版本控制
- 每次理论迭代有独立 commit 和变更说明
- 白皮书和理论文件有明确的版本号和状态标记

## 二、可复用到其他项目的流程

| 流程 | 触发条件 | 复用场景 |
|------|---------|---------|
| 知识库→GitHub 批量同步 | Get笔记 KB 有大量笔记需要归档 | 日记整理、读书笔记归档、摄影评论收集 |
| 概念操作化（四栏法） | 项目中出现了需要精确定义的核心概念 | 任何需要理论化的写作项目 |
| 竞争理论对照 | 用户声称「这个想法是新的」 | 新文章、新观点、新框架的增量性检验 |
| 反例压力测试 | 框架开始膨胀或显得「什么都能解释」 | 任何理论性、观点性写作 |
| 案例编码模板 | 需要跨案例比较分析 | 历史研究、市场分析、组织诊断 |
| Agent 角色切换（反方/审稿人/编码员/整理者） | Agent 意见与用户高度一致时 | 所有写作项目 |
| 白皮书框架搭建 | 需要从灵感走向可发表文本 | 新书写作、重要报告 |
| 成长账本 | 长期项目需要追踪系统能力而非项目成败 | 任何持续 3 个月以上的项目 |

## 三、可复用 Prompt 与指令模式

### Prompt 1：概念操作化指令
```
请为 [概念名] 建立四栏定义：
1. 定义（精确含义）
2. 可观察证据（在材料中如何识别）
3. 反证迹象（出现什么说明判断可能错了）
4. 容易误判的情况（什么看起来像、其实不是）
```
**适用场景**：任何需要精确定义的概念

### Prompt 2：反方质疑指令
```
请以反方角色，找 [框架/观点/文章] 的三个漏洞：
- 一个概念层面的（定义不清晰或过于宽泛）
- 一个证据层面的（案例能否支持结论）
- 一个逻辑层面的（是否存在因果倒置或循环论证）
不要替原文辩护。
```
**适用场景**：任何观点性或论证性写作

### Prompt 3：竞争理论对照指令
```
请列出 [观点/框架] 最接近的三个已有理论，逐项回答：
1. 已有理论能解释哪部分？
2. 解释不了哪部分？
3. 新观点多解释了哪一步？
4. 新观点是否只是改了个名字？
```
**适用场景**：声称有「新发现」或「新视角」的文章

### Prompt 4：萝卜坑自检指令
```
请检查以下案例是否存在萝卜坑风险：
- 是不是为了适配框架才选这个案例的？
- 有没有同样重要但不适配的案例？
- 换成其他理论能不能得出同样结论？
- 有没有史实材料支持？
- 是不是把结果倒推成原因？
```
**适用场景**：案例研究中

### Prompt 5：尺度锚定指令
```
请明确标注本分析属于哪个尺度：
- 产品/市场层
- 组织层
- 制度/国家层
- 文明/身份层
- 个人生产系统层
不同尺度的结论不得混为一谈。
```
**适用场景**：跨层次分析

## 五、系统未被点着（待改进）

| 缺口 | 说明 |
|------|------|
| 案例证据质量 | 大部分案例缺乏一手史料引用 |
| 多项目复用 | 此流程尚未在其他项目中验证 |
| 自动化程度 | 大部分操作仍需用户发起指令 |
| 长期追踪 | 账本和反例的长期维护机制未建立 |
| 输出到公开发表 | 从白皮书框架到可发表文本的中间环节未走通 |
